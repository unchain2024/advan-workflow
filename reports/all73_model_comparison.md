### 詳細

**[1] 返品の数量符号 (4件) — 設計判断の問題**

```
Claude/Gemini: qty=+1, amount=-6,440 (PDFの表示通り)
私のCSV       : qty=-1, amount=-6,440 (内部処理用に負号統一)
```

これは **どちらが正解とも言える**。月次請求書の表示形式に近いのはClaudeの方（数量空白＝符号なし、金額のみマイナス）。CSVを修正するか、抽出後に`is_return`なら符号反転するロジックを入れれば一致する。

該当ファイル:
- `0209SIM・佐藤1枚.pdf` (返品書)
- `0220アダストリア返品_岡部原田.pdf`
- `0224バロック返品_一和多.pdf`
- `0224バロック返品_星野.pdf`

**[2] バロック BJL 大型納品書 (10件) — 源PDF側の問題**

源PDFに単価情報が**手書きの青ペンでしか書かれていない**ため、抽出器は0円として読む。
量産品で数量の多いものは金額損失も大きい:

| ファイル | 真値 |
|---|---:|
| `0210バロックmoussyazul_星野_Part1.pdf` | ¥2,516,640 |
| `0210バロックmoussyazul_星野_Part2.pdf` | ¥2,424,200 |
| `0225バロックmoussy_星野_Part1.pdf` | ¥1,374,960 |
| `0225バロックmoussy_星野_Part2.pdf` | ¥1,422,360 |
| `0225バロックmoussy_星野_Part3.pdf` | ¥1,444,400 |
| 他5件 (合計¥18,292) | ... |

**解決策**: 業務側で**手書き単価を「@2,040円」のような印字に切り替える** か、ファイル名に単価を埋め込んで自動読取できるようにする。

**[3] アダストリア @下代単価 (4件) — 業務ルールの問題**

源PDFには本体価格(売価) 18,000円が書かれているが、実際の取引単価はファイル名@xxxxxの下代:
- `0218アダストリア岡部@8600.pdf` → 真の単価8,600円 (源は18,000円)
- `0218アダストリア岡部@6100.pdf` → 真の単価6,100円
- `0220アダストリア岡部@7150.pdf` → 真の単価7,150円
- `0220アダストリア岡部2@7150.pdf` → 真の単価7,150円

**解決策**: アダストリア納品書の場合、ファイル名から `@\d+` を抽出して単価上書きする後処理ロジックを追加。

**[4] JUN 0227 単価未抽出 (1件) — 本物のAIミス**

```
0227JUN_原田.pdf:
  doc_subtotal: 1,535,200 (フッターから読めている)
  items_sum:    0         (item行ごとの単価を読めず)
```

同じ形式の `0209JUN納前_原田.pdf` は正しく抽出できているのに、`0227JUN_原田.pdf` だけ失敗。再現性確認のため複数回走らせた方が良い。

## 推奨

1. **メインモデルは Claude Sonnet 4.6** に切替（バックエンドAPIで `EXTRACTOR_BACKEND=claude-sonnet-4-6`）
2. **業務側で改善**:
   - バロック手書き単価 → 印字単価に変更（売上¥10M超に影響）
   - アダストリア → ファイル名@単価を強制抽出するルール
3. **CSV側ロジック調整**:
   - is_return=true の数量は符号変換しない（抽出側に合わせる）
4. **JUN 0227** → 同形式の他ファイルでも再現性テスト、必要ならプロンプト改善
